If you're not familiar with rogaining, I think it's more commonly known as orienteering, but there might be slight differences between the two. For me, it means getting the map you see below, figuring out with some mates what the 'optimal' path would be to get as many controls as possible inside the allotted time, and then stumbling around the port hills in the dark for two hours before inevitably getting beaten by most of the other teams.

What really rubs is that our team usually has clocked more kilometers than alot of the teams that beat us, which means, in theory, that we're leaving a lot of points on the table. I like running as much as the next guy, but if I'm going to have to run, at the very least I want to be running efficiently.

In the absence of any real navigational prowess, the easiest option I could come up with was to get a computer to figure out the best path for me. I've already proved through years of trial and error at this point that my internal algorithm is not optimal, otherwise I'd obviously be winning more rogaines. As previously mentioned, I'm a servicable runner, probably more through being young than anything else, but there's some real sickos out there, and I'm never going to be able to run faster than them or for longer. But how well could I do if I could calculate the most optimal, most efficient path?

The program starts by importing a picture of the map of the rogaine. I then 'localise' this to easting and northing coordinates by marking some datums on the map (for example, a distinctive intersection or elevation point). The datum points are then used to get the map image's scale in meters per pixel. I can then mark all the controls on the map, and by comparing these locations to some LIDAR data for the port hills, get an accurate elevation for each of the controls.

Armed with this info, it's trivial to cycle through all the controls, and construct a graph containing, for each control, a distance to every other control, which is a prerequisite for most popular shortest-path algorithms. But it isn't quite that simple, as I don't want to find the shortest path - I want to find the most efficient path, and here is the bulk of the actual work, I think - figuring out how to describe to a computer what the most optimal path actually is

Here's an obviously sub-optimal path, where the algorithm has skipped straight to a far and very high altitude control, which (I think) is partially due to the heuristic for picking the next control not being implemented properly, and partially due to the map localisation not quite lining up right, causing some elevations to be wrong. Back to work!
